### US-D04：設計等候名單表單狀態稿

**作為** 負責 MVP 圖面設計的設計師  
**我想要** 將等候名單表單的各種互動狀態畫成元件稿  
**以便** 開發 Sprint 1（US-04）與 Sprint 2（US-05 成功／失敗 UI）時狀態切換有明確對照

**輸入格式**：
- US-D01 表單元件樣式
- [`docs/spec.md`](../../spec.md) §6.2 表單欄位與成功文案
- 錯誤文案慣例（Email 空白或格式錯誤、API 失敗繁中提示）

**輸出格式**：
- 一張 **Form States** 設計稿，建議並排 5 種狀態：
  1. **Default**：姓名、Email（必填）、身分、產品想法、卡關點、訪談意願（預設未勾）
  2. **Validation Error**：Email 未填或格式錯誤，欄位下方繁中提示
  3. **Loading**：送出中，按鈕 disabled，避免重複提交
  4. **Success**：表單區塊替換為 spec §6.2 感謝文案（不跳頁）
  5. **API Error**：顯示「表單送出失敗，請稍後再試」或同等可讀繁中訊息

**驗收條件**：
- [ ] Default 狀態含 spec §6.2 全部欄位，Email 標示必填
- [ ] Validation Error 有欄位級繁中錯誤提示樣式
- [ ] Loading 狀態按鈕不可重複點擊的視覺表現
- [ ] Success 文案與 spec §6.2 一致或標註引用 spec 原文
- [ ] API Error 有全區或表單級錯誤提示，頁面不暗示整頁崩潰
- [ ] 五種狀態皆引用 US-D01 表單元件
- [ ] 稿面標註對應實作 US：US-04（Sprint 1）、US-05 成功 UI（Sprint 2）

**依賴關係**：
- US-D01：元件與 Design Tokens
- US-D02 或 US-D03：表單在 Landing 中的嵌入位置（建議已定位）

**優先級**：P0  
**相關功能**：等候名單表單（spec §6.2、§8 訪客前台）

**備註**：
- US-05 為後端寫入，無額外畫面；成功／失敗 UI 皆在本稿涵蓋。
- 若僅實作 Sprint 1 + Sprint 2，本 US 為**最少必做**。
