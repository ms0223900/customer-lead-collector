### US-05：系統會儲存等候名單資料

**作為** 管理者  
**我想要** 系統儲存訪客填寫的等候名單資料  
**以便** 後續可以聯絡潛在學員

**輸入格式**：
- US-04 驗證通過後的表單 payload（Server Action 或 Route Handler 邊界再次驗證）
- Supabase `leads` 資料表（spec §7）

**輸出格式**：
- 一筆寫入 `leads` 的紀錄，欄位對應 spec §7：
  - 必填：`email`, `interview_intent`, `status`（預設 `新名單`）, `created_at`
  - 選填：`name`, `role`, `product_idea`, `pain_point`, `source`
- API / Server Action 回傳成功或失敗狀態給前端

**驗收條件**：
- [ ] 表單送出成功後，系統在 `leads` 建立一筆資料
- [ ] 至少儲存姓名、Email、身分、建立時間
- [ ] 產品想法、卡關點、訪談意願若有填寫，一併儲存
- [ ] 新名單預設 `status` 為 `新名單`
- [ ] 重新整理或重開後台，資料仍存在於資料庫
- [ ] 無效 Email 或必填缺失時**不寫入** Supabase
- [ ] 同一 Email 重複提交：MVP 可先允許重複，行為須與 spec 一致並可追蹤

**依賴關係**：
- US-04：表單 UI 與 client 端驗證
- Sprint 0：`leads` 資料表已建立

**優先級**：P0  
**相關功能**：名單資料模型（spec §7）、核心流程步驟 7–8（spec §5）
