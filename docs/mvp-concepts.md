# 客戶名單收集系統 MVP 核心概念與教學定位

本文件摘錄自產品規劃的核心概念，作為開發「客戶名單收集系統」時的設計與教學參考。

## 1. 核心定位

本專案的展示主題固定為：
> **《聊著聊著，AI 就做完了：不懂程式也能用 AI 做出可上線產品》課程銷售與名單收集**

它不是一個抽象的行銷工具範例，而是以真實課程推廣作為題目的實戰專案：
- **銷售標的**：AI Side Project / 不懂程式也能做可上線產品的實戰課程。
- **主要目標**：在課程正式推廣前，先用 Landing Page 收集潛在學員名單，藉由數據判斷市場熱度。
- **教學定位**：示範同一套軟體開發規格流程（SDD），如何從「股票追蹤看板」無縫映射到「課程銷售與等候名單系統」。
- **產品定位**：專注於可驗證需求的 MVP，捨棄一切不必要的複雜工程與行銷整合。

## 3. 欄位與追蹤細節設計

### 名單狀態流轉
名單的 status 設計為：`新名單` ➡️ `高意願` ➡️ `已聯絡` ➡️ `已訪談` ➡️ `不適合`。
- 訪客勾選「願意接受訪談」時，系統在建立名單時，可將狀態自動標記為 `高意願`（或由管理者後台篩選）。
- 管理者聯絡後，手動將狀態往後推進，並記錄備註。

### 簡易事件追蹤的必要性
這套系統刻意不引入第三方追蹤（如 GA4），而是由自己建立的 `analytics_events` 表統計。這樣在教學展示時，可以最清晰地向學員說明：
> "Dashboard 不是魔法，它背後其實就是把使用者在瀏覽器上的行為記錄成事件，再用 SQL 在資料庫裡把事件統計成指標。"

這是一堂極佳的 Web 原理課。

## 4. 銷售頁文案 Mock 設計的用意
銷售頁文案產生器第一版不串接真實 OpenAI / Claude API。這是極為關鍵的教學策略：
- 避免影片與課程演示卡在 **API 授權失敗**、**帳戶扣款餘額不足**或 **API 響應延遲**。
- 向學員示範：如何定義 AI 功能的邊界與介面。
- 未來可以把「串接真實 AI API」拆成額外的進階 User Story，讓學員自行挑戰。
